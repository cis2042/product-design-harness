# Glossary

| Term | Meaning |
|---|---|
| Product Design Harness | A product judgment system that runs across direction, build, launch, and feedback. |
| UX3 | User Flow, Evidence Flow, and Business Flow reviewed together. |
| Operating protocol | Six gates that apply UX3 before execution. |
| UX3 Decision Kernel | The deterministic center of UX3: Validate, Reduce, Gate, and Emit one canonical review result. |
| User Flow | The target population, actor situation, current workaround, friction, affected people, and excluded groups. |
| Evidence Flow | The sensing, learning, and calibration system that separates source, signal, interpretation, counter-signal, and decision impact; human judgment is recorded separately as accountable input. |
| Business Flow | The company, project owner, issuer, value exchange, cost, authority, stakeholders, viability, and accountability. |
| Primary user | The target actor expected to operate the product for the main outcome. |
| Secondary user | An actor involved in the workflow without being the primary target. |
| Machine actor | An automated agent, model, workflow, or software component that can sense, decide, call tools, or act. |
| Beneficiary | A person or group expected to receive benefit without necessarily operating the product. |
| Affected person | A person or group changed by the product decision, including those who do not operate it. |
| Excluded group | A population intentionally or unintentionally underserved, blocked, misrepresented, or left outside the product boundary. |
| Accountable owner | The named person or team responsible for a product decision, machine actor, claim, or execution boundary. |
| Issuer | The entity that issues a credential, claim, artifact, decision, or product promise. |
| Externality bearer | A community or group that absorbs cost, risk, confusion, exclusion, or other product externalities. |
| Gate | A decision point with input, output, and escalation rules. |
| Claim | A testable statement that must hold for the product direction to proceed. |
| Claim status | assumption or verified; a verified claim must cite at least one evidence receipt id. |
| Signal | What was observed or reported before interpretation. |
| Interpretation | What the reviewer thinks the signal means. |
| Counter-signal | Evidence or reasoning that weakens a claim. |
| Decision impact | What changes in the product decision because of the interpreted evidence. |
| Observed outcome | What happened after a bounded action was taken. |
| Evidence receipt | A traceable source, signal, interpretation, freshness record, and decision impact. |
| Evidence tier | t0-t4 operational proof level that limits the size of the next action. |
| Lane evidence tier | The t0-t4 tier each lane review reports; the tie-break in weakest flow selection when lane verdicts match. |
| Proof bar | The evidence level required before the next stage. |
| Confidence level | A human-readable judgment about trust in the current interpretation. |
| Challenge round | The council round where each lane tests another lane's material premise. |
| Context pack | The handoff package that constrains execution after a Continue verdict. |
| Human Judgment Gate | The place where taste, risk, strategy, ethics, or meaning stays human-owned. |
| Human taste | A person's accountable preference about quality, feel, meaning, taste, or strategic appetite; it is not an evidence score. |
| Human judgment | A named person's accountable decision on taste, ethics, meaning, risk appetite, or final responsibility. |
| Accountable input | A recorded human-owned input that may set direction, raise a proof bar, or accept bounded risk without pretending to be observed evidence. |
| Bounded action | A next step limited by evidence tier, risk, reversibility, scope, owner, and stop conditions. |
| Stop condition | A clear condition that blocks or downgrades execution. |
| Stop reason class | The reason class (actor, safety, core_evidence, irreversible_risk, business_viability, other) required with every stop_reframe verdict. |
| Decision rule | An operational UX3 constraint with observable triggers, evidence, counter-signals, human boundary, stop conditions, and output requirements. |
| Learning loop | signal -> interpretation -> counter-signal -> decision impact -> human judgment -> bounded action -> observed outcome -> evidence update or rule update. |
